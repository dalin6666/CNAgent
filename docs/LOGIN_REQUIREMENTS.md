# 登录功能 MVP 需求文档

## 1. 目标

为 CNAgent 增加一个可用的 Web 登录入口,实现最小登录闭环。用户输入正确的账号和密码后进入系统，未登录用户不能访问系统首页。

## 2. 本期功能

1. 登录页面
   - 展示 CNAgent 名称。
   - 提供账号、密码输入框和登录按钮。
   - 密码默认隐藏，可切换显示或隐藏。

2. 登录校验
   - 用户提交账号和密码。
   - 后端从 SQLite 用户表读取用户并校验密码。
   - 密码只保存安全摘要，不保存明文。
   - 账号或密码错误时统一提示“账号或密码错误”。

3. 登录状态
   - 登录成功后创建 Session。
   - 未登录访问首页时自动跳转登录页。
   - 已登录访问登录页时自动跳转首页。

4. 系统首页
   - 展示当前登录用户名。
   - 提供“退出登录”按钮。
   - 本阶段首页仅作为登录成功后的占位页面。

5. 退出登录
   - 用户点击退出后清除 Session。
   - 清除后返回登录页，旧 Session 不能继续访问首页。

## 3. 暂不实现

- 用户注册
- 忘记密码和邮件找回
- 手机验证码
- 多因素认证 MFA
- 企业 SSO、OIDC、SAML 和 LDAP
- 角色与权限管理 RBAC
- 多租户
- 登录审计后台
- 复杂风控、验证码和设备管理
- “记住我”长期登录

这些功能后续单独设计，不影响本期登录闭环。

## 4. 页面要求

### 4.1 登录页

- 页面简洁，首屏直接显示登录表单。
- 桌面端表单居中显示，移动端宽度自适应。
- 登录请求期间禁用按钮，避免重复提交。
- 输入为空时在对应输入框附近显示提示。
- 登录失败时不区分账号不存在或密码错误。

### 4.2 首页

- 显示“欢迎，{用户名}”。
- 显示退出登录按钮。
- 不增加与登录无关的业务功能。

## 5. 推荐技术方案

为了尽快完成并与现有 Python 项目保持一致，采用服务端渲染方案：

- Web 框架：FastAPI
- 页面模板：Jinja2
- 页面样式：原生 CSS
- 数据库：SQLite
- 数据访问：SQLAlchemy
- 密码摘要：Argon2
- 登录状态：安全 Cookie Session
- 启动服务：Uvicorn

本期不引入 React、Vue、Node.js 和前后端分离工程。

## 6. 建议目录

```text
web/
  __init__.py
  app.py                 # FastAPI 应用和路由
  database.py            # SQLite 连接和初始化
  models.py              # 用户模型
  auth.py                # 密码校验和 Session 逻辑
  templates/
    login.html
    home.html
  static/
    login.css
scripts/
  create_user.py         # 创建初始用户
tests/
  test_login.py
```

SQLite 数据文件、Session 密钥和本地 `.env` 必须加入 `.gitignore`，不能上传 GitHub。

## 7. 页面路由

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/login` | 显示登录页面 |
| POST | `/login` | 校验账号密码并创建 Session |
| GET | `/` | 登录后的占位首页，未登录时跳转 `/login` |
| POST | `/logout` | 清除 Session 并跳转 `/login` |
| GET | `/health` | 返回服务健康状态 |

## 8. 数据字段

用户表只保留本期需要的字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 用户唯一标识 |
| `username` | 登录账号，唯一 |
| `password_hash` | 密码摘要 |
| `is_active` | 用户是否可登录 |
| `created_at` | 创建时间 |

## 9. 最低安全要求

即使是 MVP，也必须满足以下要求：

1. 不得在代码、数据库或日志中保存明文密码。
2. Session 密钥从环境变量 `SESSION_SECRET` 读取。
3. 生产环境必须使用 HTTPS，并启用安全 Cookie。
4. 登录错误提示不得泄露账号是否存在。
5. SQL 操作必须使用 ORM 或参数化查询。
6. `.env`、SQLite 数据文件和 API Key 不得提交到 Git。

## 10. 验收标准

1. 使用正确账号和密码可以登录并进入首页。
2. 错误账号或密码不能登录，并显示统一错误提示。
3. 未登录直接访问 `/` 会跳转至 `/login`。
4. 登录后刷新页面仍保持登录状态。
5. 点击退出后返回登录页，再访问 `/` 仍会被拦截。
6. 密码在数据库中不是明文。
7. 登录页面在桌面和手机宽度下没有遮挡或横向滚动。
8. 自动化测试覆盖登录成功、登录失败、未登录拦截和退出登录。

## 11. 实现顺序

1. 增加 FastAPI、Uvicorn、Jinja2、SQLAlchemy、表单解析和 Argon2 依赖。
2. 建立 `web/` 目录、FastAPI 应用和 `/health` 路由。
3. 建立 SQLite 用户表和初始用户创建脚本。
4. 实现密码摘要、密码校验和 Session。
5. 实现登录、首页和退出路由。
6. 完成登录页和首页样式。
7. 编写登录流程自动化测试。
8. 本地启动并在桌面和移动端浏览器验证。
